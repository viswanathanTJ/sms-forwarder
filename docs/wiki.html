SMS Forwarder

An Android app (com.viswa2k.smsforwarder) that forwards incoming SMS to another phone, to Telegram, and to an end‑to‑end‑encrypted cloud so your other devices can read them. Use the User / Developer switch above — Developer mode reveals the architecture and the exact file:line behind every action.

User mode how to use the app, step by step.   Developer mode adds architecture, data flow, source locations, backend & release internals.

1 · What it does

📩 Forward

Every incoming SMS can be re‑sent as an SMS to another number and/or posted to Telegram chats, using a custom message format.

🔐 Cloud (E2E)

SMS are encrypted on‑device and uploaded to Firebase. Only authorized devices can decrypt and read them — the server only ever sees ciphertext.

👀 Read anywhere

Approved devices view another device's messages live, with optional push notifications.

⬆️ Self‑update

The app checks GitHub Releases and offers to install a newer version.

2 · User guide

Turn on forwarding

  1. Open the app → grant the SMS, contacts and notification permissions it asks for.
  2. Toggle Enable Service. A persistent notification shows it's running.
  3. Pick channels: Forward as SMS (enter a destination number) and/or Send in Telegram (bot API key + chat IDs).
  4. Optional: set a Device alias and a message format — %s sender, %m message, %r device alias.
  5. Tap Save Settings.
Tip: Skip Contacts stops forwarding messages from people already in your contacts (useful to forward only OTPs/automated texts).

Cloud SMS — sign in

  1. Settings → Open Cloud SMS.
  2. Sign in with Google (recommended) or email/password.
  3. If your account is approved you go straight to the cloud inbox. If not, you'll see “Account not approved” → tap Request access; an administrator approves you, then you can sign in.

Share & read other devices' SMS

  • Upload to cloud (Settings) makes this device a sender — its incoming SMS are encrypted and uploaded.
  • Receive cloud messages + the Watch screen let you pick which permitted devices to follow and whether to get push notifications.
  • The Cloud SMS screen lists decrypted messages live. The administrator controls which device may read which.

Administrator

  • The super‑admin sees an Admin button: manage the allow‑list, approve/deny access requests, grant “reader → source” access, revoke devices, and delete cloud messages.

Updates

The app checks for a newer release automatically. When one is available, an Update available: vX indicator appears under Settings → About (and a launch prompt) with an Update button that downloads + installs it — no manual checking, and the indicator clears itself once you're on the latest version.

Sign out

Use Sign out in the Cloud SMS screen's top bar to switch accounts. If your account isn't approved, the “Account not approved” screen also offers Use a different account.

3 · Architecture dev

Single‑module Kotlin + Jetpack Compose app (minSdk 29 / target 34). No DI framework; state flows through DataStore preferences, a foreground service, and a set of repositories over Firebase. New cloud code lives under app/src/main/java/com/viswa2k/smsforwarder/cloud/** (crypto, data, ui, fcm, update).

5 · End‑to‑end encryption dev

Envelope (hybrid) crypto. The server never holds a usable key.

  • Identity keys: each reader/admin device has an HPKE key pair (Tink, DHKEM(P‑256)+HKDF‑SHA256+AES‑256‑GCM). The private keyset is sealed at rest by an Android Keystore master key; the public key is published on the device's Firestore doc. Senders hold no private key.
  • Per message: a random AES‑256‑GCM Data Key (DEK) encrypts the SMS JSON; the DEK is HPKE‑sealed to each authorized recipient's public key.
  • Fan‑out: one copy per recipient at inbox/{recipientDeviceId}/messages/{messageId} (shared messageId). Admin + the source device itself are always recipients.
  • Access control = crypto + rules: an unauthorized device gets no wrapped DEK and Firestore rules deny the read.
  • Associated data (v1.1.6+): the body AEAD binds source|messageId and each wrapped DEK binds source|messageId|recipient, so a stored message can't be replayed or re‑attributed. DEKs are zeroed after use.
  • Key rotation: identity key auto‑rotates ~every 30 days on launch (CloudViewModel.onAuthenticatedInternal → CryptoManager.rotateIdentityKey()); old key versions stay in the keyset so historical messages still decrypt.

7 · Release & CI dev

  • Cut a release: uv run scripts/release.py (interactive: confirm tag+changelog → confirm push) or -y for automatic. It refuses if HEAD is already the latest tag.
  • Pipeline: pushing a v* tag runs .github/workflows/release.yml → decodes google-services.json + keystore from secrets → builds a single signed sms-forwarder-<version>.apk → publishes a GitHub Release with the generated changelog.
  • Signing: the shared keystore viswa2k.jks (alias viswa2k); its SHA‑1/256 are registered in Firebase so Google sign‑in works.
  • Secrets: GOOGLE_SERVICES_JSON, GOOGLE_WEB_CLIENT_ID, RELEASE_KEYSTORE_BASE64, RELEASE_STORE_PASSWORD, RELEASE_KEY_ALIAS, RELEASE_KEY_PASSWORD.
  • In‑app updater consumes these releases via the public releases/latest API.
